# PromnyAI ブログ執筆スタイルガイド

最終更新: 2026-05-13
適用範囲: `/blog/` 配下のすべてのブログ記事
非適用: `/docs/` 配下のマニュアル（手順書のため、現状の淡々とした書き方を維持）

## このガイドの目的

AIが書いた標準的な文章は、内容は合っていても「他人事」に読まれる。
PromnyAIブログでは、読み手が冒頭で離脱せず、最後までスクロールし、
「自分のことを言われた」と感じる記事を出すための共通ルールをここに置く。

新規記事は必ずこのガイドを通してから公開する。
既存記事は別PRで1本ずつ「冒頭3行 / 小見出し / 具体例」を中心に改修する。

## 前提：「自社の声」は3要素でできている

- **語彙**：その会社が普段使う単語、専門用語のレベル、敬語の濃さ
- **視点**：誰の側に立って語っているか（お客様 / 経営者 / 担当者）
- **温度**：感情の出し方（事実だけクール / 熱量込み / フランク）

ブログ記事はこの3要素が読み手に伝わって初めて「自社らしい」と感じてもらえる。
構成・文体ルールはすべてこの3要素を出すための手段。

## 構成ルール

### 1. 冒頭3行ルール【最重要】

ブログ記事の離脱は冒頭3行で8割決まる。次の3要素を必ず入れる。

| 行 | 役割 |
|---|---|
| 1行目 | 読み手の「あるある」を1つ提示する |
| 2行目 | その「あるある」が起きると何が困るかを1つ書く |
| 3行目 | 本記事で扱う問いを1つに絞る（解決策をまだ出さない） |

#### 良い例

> 「AIに書かせた記事、内容は合っているのに、なんだかうちの言葉に聞こえない」——AIを使い始めた経営者やマーケ担当の方からよく聞く声です。事実関係は問題ない。誤字もない。それでも自社のサイトに載せると違和感が残る。
>
> この違和感の正体は、文章の「声」です。

#### 悪い例（既存記事にありがちなパターン）

> AIで文章を書く方法について解説します。本記事では、〇〇のメリットや使い方を紹介していきます。

→ 宣言で始めると読み手は「自分ごと」だと認識せず、3秒でタブを閉じる。

### 2. 1記事1テーマ・1主張

1本のブログには **1つの問いと1つの答え** しか入れない。
「ついでに〇〇も解説します」が出てきたら別記事にする。
読み手は「結局何が言いたいのか」が分からない時点で離脱する。

### 3. 小見出しは「答え」ではなく「問い」または「具体」

#### 悪い例（断定・抽象）
- 「AI活用のメリット」
- 「重要なポイント」
- 「効果的な方法」

#### 良い例（問い・具体）
- 「では、何を捨てるか」
- 「『うちっぽい』が消える3つの瞬間」
- 「ステップ1：自社情報を『文章として』登録する」

問いの形にすると、読み手は答えを探して下にスクロールする。
これが「読まれる記事」の物理学。

### 4. まとめは5点以内 + 最後に1文の余韻

- 箇条書きは **最大5点**。それ以上は本文で語り切れていない証拠
- まとめの直後に1文、本文の核を凝縮した余韻の段落を置く
- 「ぜひお試しください」では締めない（後述「禁則ワード」）

### 5. 文末CTAは「自然な続き」として置く

- CTA枠は記事末尾に1つ。本文中に押し売りを入れない
- CTA文言は「お申し込み」「資料請求」など機能的にする
- CTA前に「PromnyAIを試してみる」という見出しブロックで囲む（既存記事と統一）

## 文体ルール

### 1. 一文を短く

- **目安：30〜50字、最大でも70字** で1文を区切る
- 「〜なのですが、〜であり、〜なため、〜となっています。」のような長文は禁止
- 接続詞を減らし、文を切って並べる

#### 悪い例
> AIで生成された文章は、事実としては正しい場合が多いものの、自社のトーンに合わないと感じることが多く、結果として手直しに時間がかかってしまうため、効率化の効果が薄れてしまうという課題があります。

#### 良い例
> AIが書いた文章は、事実としては正しい。それでも自社のトーンに合わない。手直しに時間がかかり、結局効率化にならない——これがよく聞く悩みです。

### 2. 具体名・数字・セリフを最低3か所

抽象だけの記事は読まれない。本文中に必ず以下を散らす:

- **固有名詞**：商品名、サービス名、地名、職種など
- **数字**：価格、件数、時間、割合など
- **セリフ**：「〇〇さんが言っていた」「お客様からよく聞く」など短い引用

「具体名・数字・セリフを3か所以上入れたか？」を公開前に必ず数える。

### 3. AI出力の禁則ワード集

AIの標準出力に頻出する以下の表現は、原則として **使わない**。
やむを得ず使う場合も、別の言い方を先に検討する。

#### 締めの常套句（特に厳禁）
- 「ぜひお試しください」
- 「いかがでしたでしょうか」
- 「皆様の〇〇に役立てれば幸いです」
- 「〇〇の参考になれば幸いです」

#### 断定を避けるための逃げ
- 「〜と言えるでしょう」
- 「〜と考えられます」
- 「〜と言っても過言ではありません」

#### 冗長な定型
- 「〜することができます」 → 「〜できます」
- 「〜という形になります」 → 「〜です」「〜になります」
- 「〇〇の重要性について」 → 「なぜ〇〇か」「〇〇が要る理由」

#### 大げさ・抽象
- 「驚くべきことに」「実に興味深い」「画期的」「革命的」
- 「現代社会において」「昨今の〇〇情勢を踏まえ」
- 「以下のようなメリットがあります」 → メリットを直接書く

#### 押し付けがましい煽り
- 「実は〇〇なのです」（文脈なしの単発使用）
- 「知らないと損する」「絶対にやってはいけない」
- 「〇〇すべきです」（読み手の判断に任せる表現に変える）

### 4. 段落のリズム

- **1段落は3〜5文以内**。長い段落は壁になって読まれない
- 段落の間に1行空ける（HTMLでは `<p>` を分ける）
- 強調は `<strong>` で本当に大事な1〜2か所だけ。多用すると効果が消える

## 仕上げルール

### 1. 人の手で1文だけ「現場」を入れる

AI生成の本文に、必ず1か所、AIには絶対に書けない一文を人の手で加える。

例:
- 「先週、◯◯さんが来店された際に話してくれたのですが」
- 「先月の繁忙期は本当に大変で」
- 「これは私個人の体験ですが」
- 「実際に試してみたら3日で結果が出ました」

その瞬間の現場でしか出てこない一文が1か所あるだけで、
読み手の「他人事」感が一気に消える。**この1文を入れない記事は公開しない**。

### 2. 公開前チェックリスト

公開前に必ず以下5項目を目視で確認する:

- [ ] 冒頭3行が「あるある → 困りごと → 問い」になっている
- [ ] 小見出しが「断定の名詞」ではなく「問い」または「具体」になっている
- [ ] 本文に具体名・数字・セリフが3か所以上ある
- [ ] 禁則ワードが残っていない（特に「ぜひお試しください」「いかがでしたでしょうか」）
- [ ] 人の手で加えた「現場」の1文が入っている

## 改善PRの進め方（既存記事）

既存ブログ記事の改修は **本文の意味を変えない範囲の手術** に留める。理由はSEO上の安定性。

| 触る | 触らない |
|---|---|
| 冒頭1段落 | URL（変えない） |
| 小見出しの言い回し | H1（変えない） |
| 例示・固有名詞・数字を足す | 構成全体の組み替え |
| 禁則ワードの差し替え | 既存の内部リンク |

1記事につき1PR。「diff を読みやすく、改善ポイントを明示する」ためにまとめて改修しない。
