# コードレビュー

## 使い方
ユーザーが `/code-review` と入力したとき、このスキルを実行する。

## 手順

1. `git diff` で変更差分を取得する
2. 以下のチェックリストに基づいて各変更ファイルをレビューする
3. 結果を MUST / SHOULD / NICE の3段階で分類して報告する

## チェックリスト

### MUST（修正必須）
- [ ] セキュリティ: SQLインジェクション、XSS、コマンドインジェクション等のOWASP Top 10
- [ ] 秘密情報: APIキー・パスワード・トークンがハードコードされていないか
- [ ] エラーハンドリング: ユーザー入力・外部API呼び出しにバリデーションがあるか
- [ ] データ整合性: DB操作でトランザクションが適切に使われているか
- [ ] 既存テスト: 変更によって既存テストが壊れていないか

### SHOULD（強く推奨）
- [ ] 命名: 変数・関数・クラス名が意図を正確に表しているか
- [ ] 単一責務: 1つの関数/コンポーネントが1つの責務に集中しているか
- [ ] 重複排除: 同じロジックが複数箇所にコピーされていないか
- [ ] パフォーマンス: N+1クエリ、不要な再レンダリング等がないか
- [ ] テストカバレッジ: 新規ロジックにテストが追加されているか

### NICE（あれば良い）
- [ ] 可読性: 複雑な箇所にコメントがあるか
- [ ] 一貫性: 既存コードのスタイル・パターンに合っているか
- [ ] エッジケース: 境界値・null・空配列等の考慮があるか

## 出力形式

```
## コードレビュー結果

### MUST（修正必須）
- ファイル:行番号 — 指摘内容と修正案

### SHOULD（強く推奨）
- ファイル:行番号 — 指摘内容と修正案

### NICE（あれば良い）
- ファイル:行番号 — 指摘内容

### 総評
変更の品質についての1-2行のサマリー
```
